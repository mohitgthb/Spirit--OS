01 Context and goal

An adaptive desktop for real access.

Problem

Most operating systems still assume precise mouse control, keyboard fluency, clear vision, and strong menu memory.

Response

SpiritOS keeps the familiar desktop, then adds gesture, voice, gaze, sign input, TTS, profiles, and guarded file APIs.

No installation Modern browser Real files Adaptive
Research claim: accessibility belongs in the operating-system shell, so every app inherits the same input, feedback, and safety model.

05 Hardware-free input modalities

Five input modes, one desktop

Gesture vision

MediaPipe hand landmarks plus YOLO object pass classify gestures with 800ms hold and 2s cooldown.

Voice control

Web Speech API runs 30+ direct commands; unmatched speech routes cleanly to AI agents.

Eye tracking

FaceLandmarker iris points, 9-point calibration, EMA smoothing, and dwell click support cursor control.

Sign language

TensorFlow.js reads 21 hand landmarks as 63 normalized features with dwell confirmation.

Keyboard remains available as a reliable fallback for demos, browser limits, and mixed accessibility needs.

DFS filesystem workflow

Start from the current directory rather than the root.
Sort entries alphabetically at each level.
Use cycle detection and FS_ROOT boundary checks.
Reuse a 5-second cache for listing and search.

Evidence framing

Build verifiedVite production build completes locally.
Claim scopeDFS uses sorted traversal and cache; formal benchmark study is future work.
Guarded APIsZod, rate limits, and FS_ROOT boundaries protect file routes.
Outcome: natural language becomes a bounded OS command, file operation, knowledge answer, or assistant reply.
